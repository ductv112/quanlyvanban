## Decisions Made

**1. Tách 2 file seed (001 required + 002 demo) thay vì 1 file seed_full_demo.sql mono**
Lý do: Production deploy (khi KH duyệt) chỉ cần 001 để admin login + menu render + provider seed. Dev/test env chạy cả 2 để có 50+ VB đến cho UI test. File cũ 816 dòng mono trộn admin với 7 VB đến — không rõ ràng đâu là bắt buộc, đâu là demo.

**2. KHÔNG dùng TRUNCATE/DELETE trong seed files**
Lý do: Production-safe. Nếu sau này seed 001 được chạy trên DB đã có data thật (ví dụ upgrade từ v1.0 sang v2.0), TRUNCATE sẽ phá. Chỉ dùng INSERT ... ON CONFLICT DO NOTHING / WHERE NOT EXISTS.

**3. Seed data KHÔNG upload thật MinIO**
Lý do: 35 attachments chỉ cần `file_path` string để UI render list. Upload thật 35 file PDF mỗi lần reset DB = chậm + phụ thuộc MinIO sẵn sàng. Test nghiệp vụ khác (sign flow, preview) dùng file user upload thật khi cần.

**4. Guard ở đầu seed 002 (DO block check admin user tồn tại)**
Lý do: Nếu user quên chạy seed 001 và chạy seed 002 trước → seed 002 sẽ lỗi FK constraint violation ở giữa chừng (ví dụ `created_by=1` reference admin không tồn tại). Guard raise exception rõ ràng "Chạy seed 001 TRƯỚC" giúp user debug nhanh.

**5. Mix status đầy đủ cho handling_docs (0→4 + -1 + -2)**
Lý do: HSCV có 7 status (dự thảo, đang xử lý, trình ký, chờ duyệt, hoàn thành, thu hồi, hủy). Nếu chỉ seed 1-2 status → UI HSCV page chỉ hiển thị 1 màu Tag, không test được filter/tab "Đã hủy"/"Đã thu hồi". Seed đầy đủ 7 bucket (3+3+3+2+2+1+1 = 15 records) cover mọi trạng thái.

**6. generate_series + CASE thay vì INSERT 50 lần thủ công**
Lý do: 50 INSERT rows code dài 200 dòng, khó maintain. `generate_series(1, 50) + CASE (n % 5)` tạo 50 record phong phú trong 20 dòng, dễ tăng/giảm số lượng bằng cách đổi tham số.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Schema mismatch] Plan giả định cột không tồn tại trong bảng thực tế**

- **Found during:** Task 2 khi đọc `\d edoc.*` verify column names
- **Issue:** Plan 11.1-02 viết các cột không có trong schema master v2.0:
  - `user_incoming_docs`: plan dùng `role`, `assigned_date` → thực tế chỉ có `is_read`, `read_at`
  - `attachment_*`: plan dùng `file_mime_type`, `uploaded_at`, `uploaded_by` → thực tế là `content_type`, `created_at`, `created_by`
  - `outgoing_docs`/`drafting_docs`: plan dùng `signer_id` → thực tế `signer` là VARCHAR (không FK)
  - `handling_docs`: plan dùng `code`, `progress_percent`, `due_date` → thực tế là `name`, `progress` (smallint 0-100), `end_date`
  - `inter_incoming_docs`: plan dùng `lgsp_doc_id`, `number` → thực tế là `external_doc_id`, `source_system` (không có `number`)
  - `drafting_docs`: plan dùng `create_date`, `status`, `approver_id` → thực tế dùng `received_date`, `approved` (boolean), `is_released`
- **Fix:** Đọc `\d` cho 8 bảng trước khi viết INSERT, dùng đúng tên cột schema thực tế, giữ nguyên semantics (mix status/urgent/secret/type)
- **Files modified:** `e_office_app_new/database/seed/002_demo_data.sql` (khi tạo mới đã dùng đúng schema)
- **Verification:** seed 002 apply thành công 312 records, `INSERT 0 N` đúng số lượng expected
- **Documented trong:** section "Decisions Made" của summary này (để Plan 11.1-03 cập nhật deploy scripts)

**2. [Rule 2 - Missing security check] Auth gate cho app.signing_secret_key**

- **Found during:** Task 1 viết seed 001 section 9 (signing_provider_config)
- **Issue:** Plan chỉ nói "dùng pgp_sym_encrypt với session variable app.signing_secret_key" nhưng không validate. Nếu user quên `SET`, seed sẽ fail ở giữa chừng với message khó hiểu (`current_setting returned null`).

- **Files modified:** `e_office_app_new/database/seed/001_required_data.sql`
